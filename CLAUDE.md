@AGENTS.md
- **Next.js (App Router):** Use Next.js with the App Router for routing and server rendering. Web apps need a way to decide what shows up when someone visits a page, and Next.js turns your React components into real pages a server can send to browsers. The App Router organizes everything into folders where each folder becomes a URL path. Use it for all routing and never mix in the legacy routing system. Keep route logic, layout, and server-side operations separated into their own files within each route folder.

- **TypeScript (Strict Mode):** Use TypeScript in strict mode for type safety across the entire codebase. Code breaks when you pass the wrong type of data to a function, like handing someone a phone number when they asked for an email. TypeScript catches these mistakes before your code ever runs by making you label what type every piece of data is. Enable the strictest compiler settings available. Never opt out of type safety. If you don't know the type, figure it out and narrow it down rather than escaping the type system.

- **shadcn/ui:** Use shadcn/ui for all base UI components. Building buttons, modals, and dropdowns from scratch wastes time and creates inconsistency. shadcn/ui gives you pre-built components that live in your own codebase so you can modify them. Always install components through the CLI rather than manually copying code. Keep all UI primitives in a dedicated directory and import from there so every part of the app pulls from one source of truth.

- **Tailwind CSS:** Use Tailwind for all styling. Styling normally means jumping between markup and separate style files, which splits your attention. Tailwind puts styles directly on elements using short utility classes so you see structure and style in one place. Use utility classes exclusively. Never write custom styles unless Tailwind literally cannot express what you need. Order classes logically: layout, then spacing, then typography, then color.

- **Drizzle ORM:** Use Drizzle for all database access and schema management. Writing raw query strings inside your code is error-prone and hard to refactor. Drizzle lets you define database tables as typed objects and query them with functions your editor can autocomplete and type-check. Organize schemas by domain with one file per table or entity group. Use the built-in migration tooling. Always derive application types from the schema so the database and app code never drift apart.

- **Zod:** Use Zod for runtime validation of all external data. Data coming from users, APIs, or forms could be anything, and your code crashes when it doesn't match what you assumed. Zod lets you define the exact shape data must have and rejects anything that doesn't fit before your code touches it. Create validation schemas for every external input: forms, API responses, URL params, and config values. Place schemas next to the code that uses them. Infer TypeScript types from the schemas so you never define the same shape twice.

- **Environment Validation:** Use a Zod-based environment validation library for all config and secret access. Reading environment variables raw gives you untyped strings with zero validation, and if someone misspells a variable or forgets to set one, the app breaks at runtime with a confusing error. Validate all variables at build time in a single dedicated file. Import from that file everywhere. Never read environment variables directly anywhere else.

- **ESLint + TypeScript Plugin:** Use ESLint with the TypeScript plugin for static code analysis. Linting scans your code for patterns that commonly cause bugs, like unused variables, missing return types, or inconsistent imports. Use the framework-recommended config as your base, enable strict TypeScript-aware rules, and make sure formatting rules don't conflict with your formatter. Run the linter on every save and every commit.

- **Prettier:** Use Prettier for all code formatting. Teams waste time arguing about formatting opinions like tabs vs spaces or where to put braces, and Prettier ends the argument by reformatting all code to one consistent style automatically. Configure it once with your preferences and never think about it again. The agent should never manually format code. The formatter handles it on save and on commit automatically.

- **pnpm:** Use pnpm for all dependency management. Some package managers let packages secretly use other packages they didn't declare, which causes "works on my machine" bugs. pnpm prevents this by isolating dependencies strictly so every package can only access what it explicitly lists. Use pnpm for all dependency operations. Never mix in commands from other package managers within the same project.

- **Vitest:** Use Vitest for all unit and integration testing. You need to verify your code does what you think it does without clicking through the app manually every time. Vitest runs test files that call your functions and check the outputs against expected values, and it understands your TypeScript config and path aliases out of the box. Place test files next to the code they test. Prioritize testing pure functions and server-side logic first since they give the most value per line of test code.

- **Husky + lint-staged:** Use Husky and lint-staged for automated pre-commit checks. Developers and agents forget to run the linter before committing, and bad code slips into the repository. Husky installs Git hooks that run commands automatically before every commit, and lint-staged makes those hooks fast by only checking changed files. Set up a pre-commit hook that runs formatting and linting on staged files. This guarantees no failing code enters the repository regardless of who or what wrote it.

- **Structured Logger:** Use a structured JSON logger for all application logging. Unstructured print statements create text that is hard to search and filter when something breaks in production. A structured logger outputs JSON with timestamps, severity levels, and context so you can feed it into any log analysis tool. Create a single logger instance in a shared utilities location and import it everywhere. Use leveled methods for info, warnings, and errors. Never use raw print statements in production code.

- **Responsive Design:** Use a mobile-first approach for all layouts and components. Agents default to building for desktop screens and then awkwardly squeezing things down for smaller devices, which creates broken or unusable mobile experiences. Start with the smallest screen size and scale up using responsive utility classes and dynamic sizing. Every component should work across screen sizes without horizontal scrolling or overlapping elements. Never hardcode widths or heights when relative or fluid values would work.

- **Error Handling with Blame Attribution:** Use a centralized error handling strategy that classifies every failure by its source. When something breaks, the user and the developer both need to know whether the problem is the client, the server, the network, or a third-party API. Map failures to HTTP status ranges and surface human-readable explanations that clearly state what went wrong, who is responsible, and what the user can do about it. Never swallow errors silently or show generic fallback messages. Every failure path should be intentional and informative. Wrap major sections of the UI in error boundaries so that a rendering failure in one component does not crash the entire page. A bug in a sidebar widget should show a fallback in that widget, not a blank white screen. Isolate failures to the smallest possible blast radius.

- **React Performance and Data Patterns:** Use a dedicated data fetching and caching library for all server state management. Fetching data in raw effects with manual loading and error states creates duplicated logic and poor user experience. A caching layer gives you automatic retries, deduplication, background refetching, and stale-while-revalidate behavior out of the box. Separate server state from client state. Memoize expensive computations and avoid unnecessary rerenders. Colocate state as close to where it is used as possible rather than lifting everything to the top of the component tree.

- **Accessibility:** Use semantic HTML and accessible interaction patterns for all components. Screen readers, keyboard-only users, and assistive devices rely on proper element choices and focus management to navigate your app. Use the correct semantic element for its purpose rather than styling divs to look like buttons or links. Maintain logical heading hierarchy, visible focus indicators, and keyboard navigation on every interactive element. Never remove focus outlines without providing an alternative. Test that every flow can be completed without a mouse and that screen readers can announce all meaningful content.

- **SEO:** Use descriptive metadata, structured content, and accessible media for all public-facing pages. Search engines and social platforms read your meta titles, descriptions, Open Graph tags, and canonical URLs to decide how to rank and display your pages. Every image and media element must have descriptive alt text that communicates what it shows, not just its filename. Use semantic heading hierarchy so crawlers understand content structure. Generate a sitemap and robots configuration. Name files and routes with human-readable descriptive slugs rather than IDs or hashes.

- **Analytics:** Use an abstracted event tracking layer for all user interaction measurement. Hardcoding a specific analytics provider throughout your codebase makes it painful to switch or add providers later. Create a single analytics utility that exposes generic methods for tracking page views, user actions, and conversion events. Call that utility everywhere and let the implementation details live in one place. Track meaningful user actions tied to business outcomes rather than logging every click. Respect user privacy preferences and consent requirements before firing any tracking events.

- **Authentication and Security:** Use a proven authentication library rather than building auth from scratch. Rolling your own session management, token handling, and password hashing introduces vulnerabilities that established libraries have already solved. Protect all sensitive routes with middleware that runs before any page logic. Store secrets server-side only and never expose them to the client. Enforce role-based access control at the data layer, not just the UI layer, so that hiding a button is never the only thing preventing unauthorized access. Validate and sanitize all user input on the server regardless of what the client already checked. Set secure HTTP headers including content security policy, CSRF protection, and strict transport security.

- **Project Structure and File Organization:** Use a feature-based directory structure that groups related files together rather than separating by file type. Putting all components in one folder, all hooks in another, and all utilities in a third forces you to jump across the entire codebase to understand one feature. Instead, colocate the component, its hook, its validation schema, its types, and its tests in the same directory. Keep a separate shared directory for truly reusable utilities, UI primitives, and configuration that multiple features depend on. Follow consistent naming conventions across the entire project. Never let the agent invent a new organizational pattern midway through the build.

- **Code Design and Function Discipline:** Write small functions that do one thing and name them after what that thing is. A function should be readable in isolation without needing to trace through the rest of the codebase to understand it. If a function needs a comment to explain what it does, its name or structure is wrong. Each function should hide meaningful complexity behind a simple interface. If splitting a function into smaller pieces does not make each resulting piece genuinely easier to understand in isolation, the split made things worse by spreading one idea across multiple locations. The goal is not short functions. The goal is functions where the reader does not need to look anywhere else to understand what is happening inside them. Keep arguments few and keep nesting shallow. Never repeat logic in two places. If you find yourself copying code, extract it into a shared function and call it from both locations. Prefer pure functions that take input and return output over functions that silently modify external state. Fail early and explicitly rather than letting bad data travel deep into the system before something breaks. Name variables and functions after their intent, not their type or implementation. Never abbreviate names to save keystrokes. Code is read far more often than it is written, so optimize for the reader every time. Depend on abstractions rather than concrete implementations. When one part of the system talks to another, it should go through a defined interface so either side can change independently. Compose small focused pieces together rather than building large units that know about everything. If removing or replacing one module forces changes across unrelated parts of the codebase, the boundaries are wrong.

- **Database Design and Query Discipline:** Design tables with clear relationships and integrity constraints from the start. Use database-level constraints for uniqueness, foreign keys, required fields, and valid value ranges so bad data cannot exist regardless of which code path writes it. Keep the schema simple and migration-friendly because on a new project the schema is the thing most likely to change. Do not over-normalize or add indexes upfront. Normalization and indexing are downstream of access patterns, and you do not know your access patterns until you have built the features that create them. Add indexes when you observe a query that needs one. Normalize when you see duplication causing real update problems. Every query should select only the fields it needs rather than pulling entire rows. Paginate all list queries rather than loading unbounded result sets. Use transactions when multiple operations must succeed or fail together. Use migrations for every schema change so nothing is done by hand and every change is reversible.

- **Visual Design Principles:** Use consistent spacing, clear visual hierarchy, and intentional contrast to guide the user's eye through every screen. The most important action or information on any page should be the most visually prominent. Group related elements together with proximity and whitespace rather than relying on borders and dividers to separate everything. Limit your type scale to a small set of predefined sizes and weights used consistently throughout the app. Maintain a tight color palette with a clear primary action color, a neutral base, and a semantic set for success, warning, and error states. Never use color alone to communicate meaning since it fails for colorblind users. Give interactive elements enough size and padding to be comfortably tappable on touch devices. Remove any visual element that does not communicate information or help the user navigate. Borders, shadows, gradients, and decorative details that serve no functional purpose are noise that competes with the actual content for the user's attention. Every screen should feel intentionally designed with breathing room, not like information was packed into every available pixel.

- **Commenting and Interfaces:** Use exported TypeScript interfaces for every data shape that crosses a boundary between components, between client and server, or between your code and an external API. These interfaces act as contracts that the compiler can enforce, so when one side of the boundary changes, every place that depends on it breaks loudly instead of silently. Write comments only to explain why a decision was made, never to restate what the code does. If a line of code looks strange or counterintuitive, a comment should explain the reasoning or the edge case that forced that approach. Never write comments that describe the code (like "fetch the data" above a fetch call) because these become noise that drifts out of sync with the code over time. If a function needs a comment to explain what it does, the function should be renamed or restructured instead.

- **State Management Strategy:** Use the simplest state location that fits each piece of data rather than defaulting to a global store or duplicating state across components. Put state that should survive a page refresh or be shareable via URL into route params and query params. Put state that only one component cares about into that component's local state. Only reach for shared state management when 2 or more unrelated components genuinely need access to the same data and neither URL state nor lifting state to a common parent solves it cleanly. Never centralize state just for convenience. The more state you hoist to the top of the tree, the more components rerender when anything changes, and the harder it becomes to understand which parts of the app depend on what. Every piece of state should have a single owner. One part of the system writes it, and other parts can only read it. When 2 different components can both update the same state, you get race conditions, stale data, and bugs that only show up when the user interacts in a specific order.

- **API Design Consistency:** Use a consistent structure for all API endpoints the application exposes. Every endpoint should return the same response shape so the frontend never needs special-case parsing logic for different routes. Pick a single casing convention for field names and use it everywhere. Return proper HTTP status codes that reflect what actually happened rather than returning 200 for everything with an error field in the body. Include meaningful error messages in error responses using the same blame attribution approach from your error handling strategy. Define shared response types that all endpoints use so the contract between backend and frontend is explicit and type-checked.

- **Dead Code and Artifact Cleanup:** Delete anything the codebase no longer uses the moment it becomes unused. Dead functions, unused variables, orphaned files, abandoned database columns, stale migrations, and outdated tests all accumulate silently and create confusion about what is real and what is leftover. Every piece of dead code is a lie that tells the next reader (or the agent on its next task) that this code matters and should be understood. The cost of keeping it is not disk space. The cost is attention spent reading, maintaining, and working around something that does nothing. When a feature gets removed or replaced, trace every file, function, type, schema column, and test that existed only to support that feature and remove all of it. Never delete a test just to make a new feature pass. A failing test means either the test caught a real problem or the test is wrong. Investigate which one before removing it. Never comment out code "in case we need it later." Version control already remembers everything. Commented-out code is not a safety net. It is clutter that signals uncertainty about whether the codebase is actually finished. Run the linter and type checker after every cleanup pass to verify that nothing living depended on what you removed.

- **Build Order and End-to-End Slices:** Build a thin working path through the entire stack before expanding any single layer. The most common failure on a new project is spending days perfecting the frontend with mock data, then discovering the backend cannot support what the frontend assumed. A tracer bullet is a real (not mocked) feature that touches the UI, the server logic, the database, and every integration point in between. It can be the simplest possible version of one feature, but it must be real and working from top to bottom. Build that first. It exposes integration problems on day 1 instead of day 12. Once the end-to-end path works, expand each layer with confidence that the pieces actually connect.

- **Abstraction Timing:** Do not build an abstraction until you have seen at least 2 concrete cases that need it. An abstraction shaped by only 1 case will be shaped wrong because you are guessing at what the second case will need. Guessing wrong means you either force the second case into a shape that does not fit or you tear out the abstraction and rebuild it. Both outcomes are more expensive than just having some duplication for a short time. When the second case appears, the right abstraction becomes obvious because you can see what the 2 cases share and what they do not. Duplicated code that you can see is less dangerous than a wrong abstraction that hides bad assumptions. This applies to utility functions, shared components, API client wrappers, analytics layers, and any other pattern where the agent is tempted to generalize from a single example.
